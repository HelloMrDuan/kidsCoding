# 中国大陆聚合支付与解锁确认设计规格

## 1. 背景与目标

当前项目已经具备首发家庭端课程闭环，但支付链路仍以单一 `Stripe` 逻辑为主，不符合中国大陆家庭用户首发上线要求，也缺少“支付成功但权益尚未到账”这一关键中间态的承接页面。

本规格定义首发阶段的支付硬化方案，目标是：

- 将支付架构从单一服务商逻辑提升为统一 `PaymentProvider` 抽象。
- 首发默认支持面向中国大陆家庭用户的聚合支付能力。
- 支持 `支付宝 / 微信 / 银行卡` 所覆盖的二维码支付场景，但项目代码不绑定具体服务商。
- 将“支付成功回跳”改为独立确认页，由该页面轮询统一订单状态，而不是直接假定权益已到账。
- 保留管理员按订单号手动重试同步支付结果的能力，避免 webhook 延迟或丢失时只能手工送权益。

首发成功标准：

1. 家长在 `PC/平板` 端可看到课程商品二维码并完成扫码支付。
2. 支付成功后，家长进入独立确认页，系统自动轮询订单状态。
3. 订单进入 `paid` 后，服务端统一发放课程权益并立即解锁正式课程。
4. 如果 webhook 延迟，家长不会直接看到“已支付但未解锁”的空白体验。
5. 管理员可按订单号手动重试同步支付结果，而不是直接跳过支付系统手工发权益。

---

## 2. 明确纳入范围

### 2.1 首发纳入范围

- 统一 `PaymentProvider` 接口。
- 现有 `Stripe` 保留为一个 provider 实现。
- 新增默认 `aggregated_cn` provider 接入位。
- 支付订单统一状态模型：
  - `created`
  - `pending`
  - `paid`
  - `failed`
  - `expired`
- `PC/平板` 端二维码支付链路。
- 独立支付确认页：`/parent/purchase/success`
- 成功页自动轮询订单状态，轮询时长最多 `60` 秒。
- 超时后的手动刷新按钮。
- 管理员按订单号手动重试同步支付结果。
- 订单状态与 `entitlements` 的统一映射。

### 2.2 首发明确不纳入范围

- 退款。
- 发票。
- 优惠券、促销、折扣码。
- 多商品、多课程包。
- 订阅制会员。
- 自动对账任务。
- 银行卡独立直连逻辑。
- 完整订单列表后台。
- 客服工单系统。
- 后台直接手工授予课程权益。

## 3. 支付架构

首发支付架构拆为 4 层：

### 3.1 PaymentProvider 抽象层

负责屏蔽具体支付服务商差异，统一暴露创建支付、解析 webhook、查询支付结果的接口。

首发至少保留两个 provider 实现位：

- `stripe`
- `aggregated_cn`

其中：

- `stripe` 用于兼容现有代码和测试环境。
- `aggregated_cn` 作为首发默认 provider，用于接入中国大陆聚合支付服务。

### 3.2 订单服务层

负责统一管理：

- 订单创建。
- provider 请求发起。
- webhook 结果落库。
- 手动重试同步。
- 订单状态映射。
- 权益发放。

前端页面不得直接依据第三方原始支付状态判断解锁结果，必须通过订单服务层读取平台统一订单状态。

### 3.3 前台支付页面层

包括：

- `家长购买页`
- `支付确认页`

页面只负责展示二维码、展示统一订单状态、轮询状态和触发手动刷新。

### 3.4 权益层

课程权益继续使用 `entitlements` 表表达。

规则固定为：

- 订单未确认 `paid` 前，不发放权益。
- 订单确认 `paid` 后，由服务端统一写入或激活 `entitlements`。

## 4. 统一订单状态模型

### 4.1 平台统一状态

首发统一使用以下订单状态：

- `created`
  - 平台订单已创建。
  - 尚未成功生成可支付二维码，或支付流程刚刚开始。

- `pending`
  - 已拿到二维码。
  - 等待家长扫码支付。

- `paid`
  - 支付已被服务端确认。
  - 权益已发放完成。

- `failed`
  - 支付明确失败，或 provider 返回不可恢复失败状态。

- `expired`
  - 二维码超时、订单关闭，或支付窗口已失效。

### 4.2 状态流转规则

正常流转路径：

- `created -> pending -> paid`

异常流转路径：

- `created -> failed`
- `pending -> failed`
- `pending -> expired`

首发不引入额外细分状态，例如：

- `awaiting_webhook`
- `reconciling`
- `cancelled`

这些状态即使在第三方侧存在，也应由订单服务层收敛到上述统一状态中。

## 5. PaymentProvider 接口

首发统一接口建议如下。

### 5.1 createPayment

输入：

- 平台订单号
- 用户标识
- 商品编码
- 商品标题
- 金额
- 成功回跳地址

输出：

- provider 名称
- provider 订单号
- 二维码内容或二维码地址
- 初始订单状态
- 二维码过期时间

### 5.2 parseWebhook

输入：

- 原始 HTTP 请求

输出：

- provider 订单号
- provider 原始状态
- 是否验签通过
- 平台应映射的统一订单状态

### 5.3 queryPayment

输入：

- 平台订单号或 provider 订单号

输出：

- provider 原始状态
- 平台统一状态
- 是否已支付
- 是否已过期
- 最近同步时间

## 6. 前台页面流程

### 6.1 家长购买页

路径：

- `/parent/purchase`

页面职责：

- 展示首发课程商品信息。
- 发起创建订单。
- 获取二维码支付信息。
- 引导家长扫码。

首发界面保持克制，只展示：

- 商品名称
- 课程数量
- 价格
- 单一二维码
- 当前订单状态

不在购买页内展示多支付通道切换器。

### 6.2 支付确认页

路径：

- `/parent/purchase/success`

用途：

- 承接支付成功回跳。
- 自动轮询统一订单状态。
- 避免家长直接回到总览页后看到“已支付但未解锁”的不一致体验。

### 6.3 成功确认页状态

页面只显示以下状态：

- `pending`
  - 文案：正在确认支付，请稍候。

- `paid`
  - 文案：正式课程已解锁。
  - 操作：
    - 去学习地图
    - 去家长查看页

- `failed`
  - 文案：支付未成功，请重新发起。
  - 操作：
    - 返回购买页

- `expired`
  - 文案：二维码已失效，请重新发起支付。
  - 操作：
    - 返回购买页

- `timeout_pending`
  - 文案：仍在确认中，稍后会自动生效。
  - 操作：
    - 手动刷新
    - 返回家长页
    - 说明“若已支付成功，系统稍后会自动完成解锁”

### 6.4 轮询策略

- 最多轮询 `60` 秒。
- 在 `paid / failed / expired` 时立即停止轮询。
- 超过 `60` 秒仍为 `pending` 时，切换到 `timeout_pending` 展示。
- “手动刷新”只重新查询平台订单状态，不直接触发支付补偿。

## 7. 服务端接口清单

### 7.1 POST /api/checkout

调整为统一下单入口。

职责：

- 创建平台订单。
- 读取默认 `PaymentProvider`。
- 调用 `createPayment`。
- 落库订单初始状态与 provider 订单信息。
- 返回用于前台展示二维码与跳转确认页的结果。

### 7.2 GET /api/payments/orders/[orderId]

新增统一订单状态查询接口。

职责：

- 查询平台订单。
- 计算是否已解锁。
- 返回统一状态给成功确认页轮询。

该接口是前台成功页和手动刷新按钮的唯一状态来源。

### 7.3 POST /api/payments/providers/[provider]/webhook

新增或重构为 provider 分发式 webhook 路由。

职责：

- 按 provider 解析 webhook。
- 校验签名。
- 更新订单统一状态。
- 在状态进入 `paid` 时发放 `entitlements`。

### 7.4 POST /api/admin/payments/orders/[orderId]/reconcile

新增管理员手动重试同步接口。

职责：

- 管理员输入订单号后触发。
- 调用 provider `queryPayment`。
- 根据查询结果更新平台订单状态。
- 如果确认已支付，则补发权益。

该接口是首发补偿入口，不允许后台直接跳过支付系统手工写权益。

## 8. 数据模型调整

### 8.1 orders 表

首发应稳定以下字段能力：

- 平台订单号
- `user_id`
- `provider`
- `provider_session_id` 或 provider 订单号
- `status`
- `product_code`
- `amount_cny`
- `qr_expires_at`
- `last_synced_at`
- `last_error_code`
- `last_error_message`
- `provider_status`
- `created_at`
- `updated_at`

说明：

- 首发仍只卖一个商品，但订单表必须按平台级结构设计。
- 订单表应能表达“已创建但未确认”“已支付但等待同步”等排障信息。

### 8.2 entitlements 表

保持现有简单结构即可：

- `user_id`
- `product_code`
- `status`

规则固定为：

- 仅在订单确认 `paid` 后写入 `active`

### 8.3 管理员补偿操作日志

首发可选新增轻量日志能力，用于记录：

- 谁触发了重试同步
- 针对哪个订单
- 重试时间
- 结果是否成功

如果本次实现范围受限，可先使用现有服务端日志，不强制要求单独表结构。

## 9. 管理员补偿流程

首发补偿方式固定为：

- 管理员按订单号手动重试同步支付结果

不支持：

- 管理员直接手动授予课程权益
- 管理员手工将订单改为 `paid`

补偿流程：

1. 管理员输入订单号。
2. 服务端读取订单与 provider 信息。
3. 调用 provider `queryPayment`。
4. 若 provider 已确认支付：
   - 更新订单状态为 `paid`
   - 发放或激活 `entitlements`
5. 若 provider 仍未支付：
   - 仅更新最近同步时间和 provider 状态
   - 返回当前结果

这样可以保证“账”和“权益”始终通过同一真相来源收敛。

---

## 10. 现有 Stripe 代码的处理原则

本规格不要求立即删除现有 `Stripe` 逻辑，而要求将其下沉为 `PaymentProvider` 的一个实现。

具体原则：

- 保留 `stripe` provider，避免破坏现有测试和开发路径。
- 将默认支付入口改为读取平台默认 provider，而不是写死 `Stripe`。
- 新增 `aggregated_cn` provider 实现位，作为首发默认选择。
- 前台页面不再直接理解 `Stripe Checkout` 会话概念。

也就是说：

- `Stripe` 从“系统主支付流程”降为“一个实现”。
- 首发主流程切换到大陆聚合支付。

## 11. 测试与验收要求

### 11.1 单元测试

至少覆盖：

- 统一订单状态映射。
- provider 抽象接口的成功 / 失败 / 过期映射。
- 成功确认页状态转换逻辑。
- 管理员手动重试同步逻辑。

### 11.2 集成测试

至少覆盖：

- 创建订单后返回二维码信息。
- webhook 到达后订单状态更新并发放权益。
- 成功确认页轮询后进入 `paid`。
- 超时后显示手动刷新与说明。

### 11.3 E2E

至少覆盖：

- 家长发起购买后进入确认页。
- 订单状态由 `pending` 变为 `paid` 后，学习地图正式课程解锁。
- 管理员按订单号重试同步后，前台确认页可刷新为已解锁。

### 11.4 首发验收标准

上线前必须满足：

1. 家长可以看到二维码并发起支付。
2. 成功页可以轮询统一订单状态。
3. 支付成功后权益由服务端统一发放。
4. `60` 秒未确认时，页面能给出清晰说明和手动刷新按钮。
5. 管理员可以按订单号手动重试同步支付结果。
6. 前台页面不直接依赖具体支付服务商状态字段。

## 12. 实施顺序建议

建议按以下顺序实现：

1. 抽象 `PaymentProvider` 接口并收敛现有 `Stripe` 逻辑。
2. 完善 `orders` 统一状态模型。
3. 改造 `/api/checkout` 为统一下单入口。
4. 新增统一订单状态查询接口。
5. 新增成功确认页与轮询逻辑。
6. 新增 provider 分发式 webhook。
7. 新增管理员手动重试同步入口。
8. 跑通支付成功、延迟和补偿三条主路径。

---

## 13. 自检结论

本规格已完成自检，结论如下：

- 无 `TODO`、`TBD` 或“实现时再决定”占位语句。
- 已明确首发纳入范围与不纳入范围，避免把退款、对账、会员等一起拉入本次实施。
- 订单状态模型、前台状态模型与管理员补偿规则保持一致，无内部冲突。
- 已明确“前端轮询订单状态，由服务端映射解锁结果”，避免直接轮询 `entitlements` 与设计冲突。
