# 中文文案修正设计规格

## 1. 背景

当前仓库已经有可运行的第一期 MVP Web 应用，但用户可见中文文案、课程内容配置和部分仓库文档存在明显乱码。这会直接影响儿童端可用性、家长理解成本以及后续维护效率。

本次工作的目标不是新增业务功能，而是完成一次有边界的中文文案治理：

- 修复应用内所有用户可见乱码
- 统一应用与文档中的简体中文表达
- 建立页面文案和内容数据的清晰归属
- 保持现有功能、数据结构和测试主流程稳定

## 2. 目标

本次设计要解决以下问题：

1. 首页、学习流程、作品完成页、家长页等用户界面可以正常显示简体中文。
2. 测评题目、课程步骤、卡片名称等内容型文本恢复为可读的简体中文。
3. 仓库内中文文档恢复为 UTF-8 简体中文，并统一术语和表达。
4. 后续新增文案时，开发者能够明确判断应放入 `messages` 还是 `content`。
5. 自动化测试尽量校验行为而不是绑定文案，避免后续改字导致测试脆弱。

## 3. 非目标

本次不包含以下内容：

- 新增课程、卡片、登录、支付等业务功能
- 引入完整多语言国际化框架
- 重构现有数据模型
- 调整路由结构
- 大规模改写测试体系

## 4. 范围

### 4.1 应用代码

需要覆盖 `apps/web` 中的以下区域：

- `src/app/*` 页面文案
- `src/features/*` 中的界面提示、按钮、标签、状态文案
- `src/content/*` 中的测评题目、课程标题、步骤提示、卡片名称与系列
- `README.md`、`apps/web/README.md`
- `docs/superpowers/` 下现有中文规格与计划文档

### 4.2 文案类型

本次处理的文案分为两类：

- 页面壳层文案：页面标题、按钮文字、页面说明、空状态、成功提示、通用标签
- 内容型文案：课程目标、步骤说明、测评题目、卡片名称、系列名称

## 5. 文案分层方案

### 5.1 页面文案层

新增文件：

- `apps/web/src/messages/zh-CN.ts`

职责：

- 集中管理用户界面壳层文案
- 为页面和组件提供统一的中文文案来源
- 统一通用按钮、状态提示、标题、副标题等表达

适合进入该层的内容：

- 首页标题与按钮
- 引导页说明
- 学习地图页面说明
- 完成页提示
- 家长查看页说明
- 通用按钮文案
- 通用状态与错误提示

### 5.2 内容数据层

继续保留在现有 `content/`：

- `apps/web/src/content/assessment/questions.ts`
- `apps/web/src/content/cards/card-definitions.ts`
- `apps/web/src/content/lessons/story-path.ts`

职责：

- 承载课程、题库、卡片等业务内容
- 保持内容数据和页面壳层文案分离
- 只修正文案值，不改变原有数据结构

适合保留在该层的内容：

- 测评题目与选项
- 课程标题、目标、步骤说明
- 卡片名称、分类、系列描述

### 5.3 文档层

文档统一要求：

- 全部使用 UTF-8 编码
- 全部使用简体中文
- 术语统一
- 保持可直接阅读，不保留乱码内容

需要处理的文档包括：

- 仓库根目录 `README.md`
- `apps/web/README.md`
- `docs/superpowers/specs/*.md`
- `docs/superpowers/plans/*.md`

## 6. 文件边界规则

为避免本次修完后再次散乱，定义以下边界：

1. 只有界面壳层文案进入 `messages/zh-CN.ts`。
2. 课程题目、课程说明、卡片名称等业务内容继续保留在 `content/`。
3. 页面组件尽量只消费文案，不直接内嵌大段中文字符串。
4. 文档不复用 `messages`，但必须沿用相同术语体系。

## 7. 术语统一

本次统一使用以下说法：

- 积木编程
- 学习地图
- 轻量测评
- 游客进度
- 绑定账号
- 家长查看页
- 卡片收集
- 作品完成页
- 连续学习

禁止在不同文件中混用同义词造成理解偏差，例如：

- “测评 / 测试 / 问答” 混写
- “家长端 / 家长页 / 家长面板” 混写
- “保存进度 / 绑定进度 / 同步进度” 无边界混写

## 8. 执行顺序

本次建议按以下顺序实施：

### 8.1 页面壳层文案

优先处理主用户路径页面：

- `/`
- `/onboarding/age`
- `/onboarding/check`
- `/learn/map`
- `/learn/lesson/[lessonId]`
- `/project/[projectId]/complete`
- `/auth/bind`
- `/parent/overview`
- `/cards`

目标：

- 先恢复用户主流程的中文可读性

### 8.2 组件与通用界面文案

处理以下区域：

- `features/onboarding/*`
- `features/lessons/*`
- `features/auth/*`
- `features/cards/*`
- `features/parent/*`

目标：

- 将分散在组件中的按钮、提示、标签、状态文本收口到 `messages`

### 8.3 内容数据

处理以下文件：

- `content/assessment/questions.ts`
- `content/cards/card-definitions.ts`
- `content/lessons/story-path.ts`

目标：

- 修复内容值中的乱码
- 保持原有结构与业务逻辑不变

### 8.4 文档

最后修复仓库中文文档，确保术语和产品内表现一致。

## 9. 实施规则

### 9.1 页面文案集中化

页面或组件如果只是在展示中文提示，应优先从 `messages/zh-CN.ts` 读取。

### 9.2 内容数据完整表达

`content/` 中的文本应尽量保持完整句子或完整标题，不在页面中用碎片化字符串拼接长句。

### 9.3 测试以行为为主

自动化测试策略如下：

- E2E 继续优先依赖 `data-testid`
- 单元测试只在确实需要时校验展示文案
- 避免把用户可见中文作为流程验证的唯一依据

### 9.4 编码规范

本次修过的中文文件统一为 UTF-8 编码，后续新增中文文件也必须遵守相同规则。

## 10. 风险与控制

### 10.1 风险：修文案时误改业务逻辑

控制方式：

- 限制改动范围在字符串和文案引用关系
- 不修改已有数据结构和路由

### 10.2 风险：文案集中后页面取值出错

控制方式：

- 主路径页面逐一验证
- 为关键页面保留现有测试覆盖

### 10.3 风险：文档修正与应用术语不一致

控制方式：

- 以本规格第 7 节术语为唯一基准

## 11. 验收标准

本次工作完成后，应满足以下标准：

1. 主用户路径页面不再出现乱码。
2. 测评题目、课程标题、步骤说明、卡片名称全部显示为正常简体中文。
3. 仓库内所有中文文档可直接阅读，不再出现乱码。
4. 页面壳层文案可从 `messages/zh-CN.ts` 统一获取。
5. `content/` 中仅保留内容型文案，不混入大量页面壳层文案。
6. 现有功能行为不变。
7. 现有自动化测试在必要调整后保持通过。

## 12. 交付结果

本次最终交付应包括：

- 一个集中管理页面文案的 `messages/zh-CN.ts`
- 修正后的页面与组件中文文案
- 修正后的内容配置文案
- 修正后的仓库中文文档
- 保持通过的测试与构建验证
