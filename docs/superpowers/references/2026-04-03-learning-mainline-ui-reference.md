# Learning Mainline UI Reference

## 1. Purpose

This reference captures the page and learning-pattern rules we will reuse during the learning-mainline redesign. It is not a visual clone guide. It records which successful product behaviors are worth borrowing and how they translate into this project.

## 2. International Kids Coding References

### Tynker

- The homepage hero leads with what children can make, not with platform plumbing.
- Learning paths are framed as progression, not as a raw syllabus dump.
- Parent trust is built through outcomes, structure, and age-fit guidance.
- Good translation for this project:
  - lead the homepage with a story-stage outcome
  - show a three-step growth path before deeper curriculum detail

### Kodable

- Self-guided learning is presented as something children can start without adult co-piloting every minute.
- Early lessons use tiny goals and obvious positive feedback.
- Interface pressure stays low: one clear next action, not many competing controls.
- Good translation for this project:
  - lesson pages should emphasize one small task at a time
  - the map should always make the next step obvious

### ScratchJr

- Story creation is treated as expression, not just puzzle completion.
- Characters, movement, dialogue, and scenes are the heart of learning.
- The interface feels playful while staying legible.
- Good translation for this project:
  - the child-facing mainline should stay rooted in animated story creation
  - the lesson shell should feel like a creation stage, not a coding console

### codeSpark

- Puzzle loops are short and always point back to making something.
- Low-friction rewards support momentum, but the stronger reward is what the child created.
- Good translation for this project:
  - keep pass/fail pressure low
  - weight rewards toward completed story output, not just checkmarks

## 3. Domestic Child Learning Product References

### Shared domestic product patterns worth borrowing

- The first screen must reassure parents quickly: age fit, learning mode, visible result.
- Conversion works better when the value story is short and concrete.
- Parent-facing surfaces work best when progress, output, and next action are grouped together.
- Visual hierarchy tends to be cleaner than feature-rich SaaS dashboards: large blocks, strong headings, fewer competing actions.

### Good translation for this project

- The homepage first screen should attract children visually, but calm parents informationally.
- Parent overview should lead with recent work and next action, not just metrics.
- Learning pages should feel premium and guided, not purely functional.

## 4. Product Translation Rules

### Homepage

- Use a story-stage hero instead of a generic education banner.
- Show what a child can create before explaining the curriculum.
- Keep the first-screen promise concrete: a child can make animated stories.

### Learning Map

- Present the map as a growth route, not a lesson table.
- Make the current step and next step visually obvious.
- Completed nodes should signal progress, not just completion status.

### Lesson Page

- Structure the page around "today's small goal".
- Keep the stage and feedback visible.
- Hints should feel supportive rather than corrective.

### Parent Overview

- Lead with recent work, growth summary, and next action.
- Show learning as a narrative of progress rather than a pile of counters.

## 5. Guardrails

- Do not copy the exact layout of any one platform.
- Do not borrow adult SaaS patterns for child-facing learning pages.
- Do not sacrifice clarity for decorative motion.
- Every visual choice must support the learning mainline: start, progress, create, review.
